Onboarding · First launch NEW

8.Permissions Setup

First-run screen after app install. Three permission cards: Notifications (required), Read SMS (optional, enables auto-import), Exact Alarms (required for precise WorkManager firing on Android 12+). Progress dots indicate step 2 of a 3-step onboarding.

08-onboarding.svg390×844

Annotations

  • 1 — Notifications toggle (ON by default). Tapping calls NotificationManagerCompat.requestPermission() on Android 13+. Badge: "Required".
  • 2 — SMS toggle (OFF by default). Tapping shows rationale and calls READ_SMS permission request. Badge: "Optional". If denied, SMS import in Reminders shows a locked state with "Enable in Settings" shortcut.
  • 3 — Exact Alarms toggle (ON). Android 12+ requires SCHEDULE_EXACT_ALARM. Toggling opens system settings deep-link if not granted.
  • 4 — "Continue" CTA navigates to Home Dashboard. "Skip for now" grants no optional permissions and also navigates home — no blocking.
Why dark background: Consistent with Splash Screen brand moment. Permission requests feel less clinical when framed as part of a deliberate setup flow rather than system dialogs.
Design lens — Ethics: SMS toggle is OFF by default (data minimization). Clear "Optional" label. Rationale shown before request. No dark pattern of pre-checking privacy-sensitive permissions.
Entry point · Launch

1.Splash Screen

Cold launch screen. App logo, name, tagline and loading indicator while Hilt DI and Room initialise. On first launch flows to Onboarding (Screen 8). On subsequent launches checks for notification permission and flows to Home Dashboard.

01-splash.svg390×844

Annotations

  • 1 — Logo: circle with ₹ rupee symbol, centred at 35% screen height. Token: color/primary (white on black).
  • 2 — App name "EMI Reminder" at text/headline-medium + tagline at text/body-medium.
  • 3 — 3-dot loading indicator at 90% height. Animates left-to-right. Hides once Room DB initialised.
Why dark background: Prevents flash of white during Android window background before Compose renders. Standard Material 3 splash pattern.
Core · Primary screen

2.Home Dashboard

Portfolio-at-a-glance. Active loan count, next due date, total monthly outflow with mini trend chart, quick action shortcuts, and 2 most urgent reminders. Tapping a reminder opens Loan Detail (Screen 10).

02-home-dashboard.svg390×844

Annotations

  • 1 — Greeting header with avatar initials, personalised name, and bell notification icon. Bell badge shows unread notification count.
  • 2 — Summary cards: 2×half-width (Total Loans, Next Due) + 1×full-width dark card (Total EMI/Month + mini bar chart). Design lens: Von Restorff — dark card draws attention to the primary metric.
  • 3 — Quick Actions row: Calculator, Add Reminder, View All. Design lens: Fitts's Law — 48×48dp minimum tap targets. Recognition over Recall — labelled icons not bare glyphs.
  • 4 — Recent Reminders: 2 cards with left-border urgency encoding. Black border = due ≤7d, grey = ≤14d, dimmed = paused. "See all →" navigates to Reminders list.
Why dashboard first: Reference app has no summary view — lands on Calculator. Users with multiple loans need a portfolio view for orientation. Design lens: Mental Models (banking app conventions), Goal-Gradient (next due creates urgency).
State · First-time NEW

16.Empty State — Dashboard

First-time user sees this instead of the normal dashboard. Illustration + dual CTAs to guide them to either add a loan manually or calculate an EMI first. SMS import shortcut surfaces the auto-import path without friction.

16-empty-state-dashboard.svg390×844

Annotations

  • 1 — Geometric illustration using the ₹ brand mark. No copyrighted images. Uses only color/outline and color/primary tokens — renders correctly in dark mode.
  • 2 — Primary CTA "Add Your First Loan" — direct path to Add Reminder bottom sheet. Full-width, black fill (color/primary).
  • 3 — Secondary CTA "Calculate EMI First" — for users who want to explore before committing. Outlined style.
  • 4 — SMS import hint card. Surfaces only if READ_SMS permission was granted during onboarding. Hidden otherwise — no hollow offers.
Design lens — Progressive Disclosure: Empty state shows only 2 primary actions, not the full app surface. Reduces cognitive load (Miller's Law) for first-time users who haven't formed a mental model yet.
Feature · Calculator

3.EMI Calculator

Loan-type tabs, large styled inputs with sliders, steppers for interest rate, and a live preview of the estimated EMI. Feeds into Results (Screen 4). Now includes Interest Type selector (Screen 17).

03-emi-calculator.svg390×844

Annotations

  • 1 — Segmented pill tabs (Home / Car / Personal). Pill vs. underline — tighter on narrow screens. Design lens: Chunking (groups related loan types).
  • 2 — Loan Amount: 28sp bold number + ₹ prefix. Slider range: ₹1L–₹2Cr for Home, ₹50k–₹30L for Car, ₹10k–₹10L for Personal. Range adapts to selected tab.
  • 3 — Interest Rate: − / + stepper at 0.1% increments. Typed value supported. Min 1%, max 36%.
  • 4 — Tenure with Years/Months toggle. Shows both units simultaneously (e.g., "20 years · 240 months").
  • 5 — Live preview card updates at 300ms debounce. Design lens: Doherty Threshold — feedback under 400ms feels instant.
Why live preview: Reference app requires explicit Calculate tap. Running estimate makes sliders interactive and reduces "did it do something?" anxiety. Debounced 300ms to avoid calculation jitter during dragging.
Feature · Calculator advanced NEW

17.Interest Type Selector

Flat Rate vs. Reducing Balance selector within the Calculator screen. Visual mini-charts inside each option card show how principal/interest behaves differently. Inline comparison warning surfaces the EMI difference to prevent user confusion.

17-interest-type-selector.svg390×844

Annotations

  • 1 — Default is Reducing Balance (most common Indian home loan method). Pre-selected with filled card + checkmark. Design lens: Defaults / Behavioral Science.
  • 2 — Option cards with inline mini bar charts. Reducing = bars decrease (interest portion shrinks). Flat = uniform bars. Visual metaphor encodes the mathematical concept without text. Design lens: Recognition over Recall.
  • 3 — Inline comparison warning: "Same rate, different EMI amounts: Reducing ₹32,450 · Flat ₹52,083". Prevents confusion when users switch type and see a 60% EMI jump. Design lens: Norman's principle of feedback.
  • 4 — Live preview updates immediately on type change. No extra tap needed.
Why this matters: Reference app doesn't expose interest type. Many personal loan and vehicle loan tiers quote "flat rate" which looks lower but costs ~60–80% more. This transparency is a competitive differentiator and an ethical design requirement.
Feature · Results

4.Calculator Results

Dedicated results screen with EMI hero card, donut chart (principal vs. interest split), 3-metric summary, and collapsible amortization preview. "View Full Schedule" navigates to Screen 11. "Set Reminder" pre-fills the Add Reminder sheet with these values.

04-calculator-results.svg390×844

Annotations

  • 1 — Hero EMI card: color/primary background, text/display-large (36sp) for the EMI amount. All input params shown in small text underneath — no hidden information.
  • 2 — Donut chart: black arc = principal %, grey arc = interest %. Centre total. Design lens: Gestalt figure/ground.
  • 3 — 3-column summary inside white card: Total Payment | Total Interest | Principal. Column dividers for visual separation.
  • 4 — "View Full Schedule" → Screen 11. "Set Reminder" → pre-fills Add Reminder bottom sheet with calculated EMI amount + loan type. Reduces re-entry friction (Tesler's Law).
Why a separate results screen: Reference app shows results inline. Dedicated screen with visual hierarchy makes the output feel as premium as a fintech app and allows sharing via native share sheet.
Feature · Advanced calculator NEW

11.Amortization Schedule

Full month-by-month amortization table. Current month highlighted with left-border accent. Year jump filter for fast navigation across 20-year loan tenures. Yearly summary card after each year's rows. Export to CSV/PDF.

11-amortization-schedule.svg390×844

Annotations

  • 1 — Loan summary bar (black, full-width) at top. Always visible — no need to scroll back to remember which loan you're viewing.
  • 2 — Current month row: highlighted with color/background tint + left-border accent + "Month 1 of 240" sub-label. Serial Position — current position grounded.
  • 3 — Yearly total card summarises all 12 months in one collapsed card. Reduces cognitive load for users scanning long-term interest costs.
  • 4 — Export button: triggers FileProvider share sheet with CSV or PDF generated via iText on-device. No server calls required.
Design decision on layout: Horizontal scrolling table (common pattern) was rejected — too many horizontal swipe conflicts with system back gesture. 4-column compact layout with right-aligned numbers (easier to scan vertically) chosen instead.
Feature · Advanced calculator NEW

12.Loan Comparison Calculator

Side-by-side comparison of two loan options. Each panel has independent Amount, Rate, and Tenure inputs. Results table shows Monthly EMI, Total Payment, Total Interest, and Duration for both. Recommendation card highlights the financially better option.

12-comparison-calculator.svg390×844

Annotations

  • 1 — Two input panels side-by-side. Option A (dark header) vs. Option B (grey header). Both panels are independent — changing one does not affect the other. Swap icon (⇄) between panels lets user copy Option A values to Option B as a baseline.
  • 2 — "Compare Results" CTA triggers calculation for both options simultaneously. Results appear below.
  • 3 — Comparison table with divider. "Better" badge appears on the winner column for each row independently (EMI winner may differ from Total Interest winner).
  • 4 — Recommendation card (black): plain-language explanation of the tradeoff — higher EMI vs. lower total interest. Design lens: Plain Language / Inverted Pyramid.
Why this matters: Users comparing fixed vs. floating rate offers from multiple banks currently use a spreadsheet or mental math. This calculator is the key competitive differentiator absent in the reference app.
Feature · Advanced calculator NEW

13.Prepayment Calculator

What-if analysis for extra loan payments. Shows months saved and interest saved from a one-time, monthly extra, or annual prepayment. Two outcome options: reduce tenure (same EMI, done earlier) or reduce EMI (same tenure, lower monthly).

13-prepayment-calculator.svg390×844

Annotations

  • 1 — Current status bar (black, full-width): shows outstanding balance + months left at a glance. Grounding context before the what-if input. Design lens: Anchoring — sets reference point for the impact comparison.
  • 2 — Prepayment type: One-time / Monthly extra / Annual. One-time is default — most common prepayment scenario.
  • 3 — Before / After comparison table with "Interest saved" highlighted row (coloured background). Design lens: Von Restorff — savings stands out.
  • 4 — Two outcome choice cards: Reduce Tenure (same EMI, earlier end) vs. Reduce EMI (same tenure, lower monthly). Card selection drives the amortization recalculation. Design lens: Commitment & Consistency.
Why this is valuable: Most Indian home loan borrowers don't realise a single large prepayment early in the tenure can save lakhs in interest. This screen makes the saving visceral and actionable — core to the "smart" positioning.

5.Loan Reminders List

Full list of all loan reminders. Search bar, urgency filter chips, colour-coded left-border cards, per-card enable/disable toggles. FAB triggers Add Reminder bottom sheet. SMS import button in toolbar for first-time setup.

05-reminders-list.svg390×844

Annotations

  • 1 — Urgent card: black left-border + black "Due in 3d ⚠" chip. Draws the eye immediately — Von Restorff effect.
  • 2 — Standard card: grey left-border + grey chip. Lower urgency, quieter.
  • 3 — Paused card: light grey border, all text dimmed. State is clear without removing the entry.
  • 4 — FAB (+): Material 3 extended FAB pattern. Sticky at bottom-right. Opens Screen 6 bottom sheet.
Urgency encoding: Black border = due ≤3d; dark grey = ≤7d; light grey = ≤14d; no accent = normal; dimmed = paused. Token-driven — easily adjustable for user-defined urgency thresholds in Settings.

6.Add / Edit Reminder

Bottom sheet modal. Chip-based loan type and frequency selectors. Large EMI amount input. Monthly due date with calendar picker. "Save Reminder" triggers WorkManager scheduling.

06-add-reminder.svg390×844

Annotations

  • 1 — Loan name free-text, auto-capitalised. Shows example placeholder text.
  • 2 — Loan type chips (Home / Car / Personal / Other). Only 4 options → chips, not dropdown. Filled black = selected.
  • 3 — EMI Amount: 28sp bold with ₹ prefix. Numeric keyboard on tap.

Feature · Import NEW

14.SMS Auto-Import Sheet

Bottom sheet triggered from Reminders → import icon. Parses last 30 days of bank SMS messages for EMI patterns. Shows detected loans as checkable cards. Import selected. Privacy-respecting: SMS parsed on-device, never uploaded.

14-sms-import.svg390×844

Annotations

  • 1 — "Select all" checkbox with count. Defaults to all checked. Design lens: Defaults — user more likely to import all than select individually.
  • 2 — Detected loan card: shows parsed bank name, EMI amount, and inferred due date. SMS snippet shown verbatim so user can verify the source. "SMS" badge distinguishes auto-detected from manual.
  • 3 — Unchecked card: Axis CC EMI deselected because it's a credit card EMI (different financial product). User may choose to exclude.
  • 4 — "Import 2 Selected Loans" CTA. Count updates dynamically with checkbox changes. No data leaves the device — SMSReceiver reads locally, inserts to Room, SMS raw content not stored.
Ethics / Privacy: SMS not stored persistently — only parsed metadata (amount, date, bank). "SMS not stored" note on-screen. Only available when READ_SMS was explicitly granted. Pattern follows Google Messages design conventions.

9.Portfolio Analytics

EMI trend bar chart (3/6/12 month or all-time), donut breakdown by loan type, 3-month summary metrics, and an insight card with proactive recommendations. Accessible from Home Dashboard → Analytics or from Loan Detail.

09-loan-analytics.svg390×844

Annotations

  • 1 — Bar chart with time period selector. Monthly totals per bar. Value labels above bars. Design lens: Information Scent — bars grow visibly over time (loan additions).
  • 2 — Donut chart by loan type. Colour-coded slices: black (Home), dark grey (Car), light grey (Personal). Legend shows amount + percentage. Tapping a slice navigates to that loan's detail.
  • 3 — Summary metrics: Total Paid, Interest Paid, Principal Paid for the selected period. Helps users understand how much of their payments are going to interest.

10.Loan Detail

Individual loan deep-dive. Dark header card with loan name, bank, EMI amount, and next due badge. Progress ring (% paid). Next payment card with inline Pay Now button. Action buttons (Edit, Amortization Schedule, Prepay). Payment history list with paid/late status.

10-loan-detail.svg390×844

Annotations

  • 1 — Dark header: loan name, bank + masked account, EMI, next due badge. Expands the brand/trust moment from Splash into individual loan context. Design lens: Visceral trust (Norman level 1).
  • 2 — Progress ring at 40%: shows how far along the loan is. Centre: "40% paid". Right panels: Principal | Total Interest + amount paid so far. Design lens: Goal-Gradient.
  • 3 — Next payment card: date, amount, inline "Pay Now" button. "Pay Now" opens bank UPI deep-link or payment reminder. Reduces friction — user doesn't need to leave the app.
  • 4 — Payment history: chronological list with "Paid" (grey chip) and "Late" (black chip) status. Provides accountability and helps user identify recurring late payment patterns.
Why payment history matters: Reference app has no payment history. Tracking late payments helps users improve their repayment habit (Commitment & Consistency) and may be useful for credit score monitoring integrations in future.

7.Notification Preview

Lock screen notification from WorkManager. App icon, title with loan name, body with amount + date. Two quick-reply action buttons: Pay Now (PendingIntent → UPI) and Remind Later (snoozes 24h via one-shot WorkManager request).

07-notification.svg390×844

Annotations

  • 1 — Frosted-glass notification card. Android 12+ system standard. App icon (₹ box) + app name + relative time. Not customisable beyond icon/content.
  • 2 — Notification title: loan name truncated to 1 line.
  • 3 — Action buttons: "Pay Now" → PendingIntent → bank UPI intent. "Remind Later" → PendingIntent → OneTimeWorkRequest with 24h delay.
  • 4 — Stacked secondary notifications for other pending loans. InboxStyle grouping in NotificationCompat.

15.Settings

Grouped settings screen: Notifications (reminder toggle, advance days, reminder time, overdue alerts), Appearance (theme, currency, language), Data (CSV export, Google Drive backup/restore), and About. All rows are tappable; toggles are inline.

15-settings.svg390×844

Annotations

  • 1 — Notifications section: master toggle + sub-settings for advance days, reminder time, and overdue alerts. Sub-settings visually de-emphasised when master toggle is OFF (greyed out).
  • 2 — Advance reminder: "3 days" (default). Tapping opens a NumberPicker dialog (1–30 days). Design lens: Defaults — 3 days pre-selected based on typical EMI processing time.
  • 3 — Appearance: Theme (System/Light/Dark), Currency (₹ INR default), Language. Theme picker → Material 3 dynamic colour theming. Currency changes the ₹ symbol app-wide via DataStore.

Design lens — Shneiderman's 8 Golden Rules: Settings groups by function (not alphabetical). Most dangerous action (data wipe) absent from this screen — not surfaced here, requires separate confirmation flow if ever added.

For reviewer

Open questions

  • Navigation structure — Current design uses Bottom Nav with 3 tabs (Dashboard | Calculator | Reminders) + Analytics accessible from Dashboard. Alternative: 4-tab nav (Dashboard | Calculator | Reminders | Portfolio). Which reflects usage frequency in the dev team's testing?
  • Urgency colour encoding — Wireframes use monochrome (black/grey/dim). High-fidelity: should urgency use colour semantics (red <3d, amber <7d, green = normal) or keep monochrome for consistency with Material You dynamic theming? Colour adds urgency but may fail for colorblind users unless icon + label also change.
  • Loan type taxonomy — Current chips: Home / Car / Personal / Other. Should "Business Loan" and "Education Loan" be first-class types (common in India)? Adding them means 6 chips — may need to wrap to 2 rows.
  • SMS auto-detect scope — Current design scans last 30 days. Longer window (90 days) catches more historical EMIs but risks finding repaid loans. Should the scanner filter by "recurring" pattern (same amount, same sender, monthly interval)?
  • Pay Now integration — "Pay Now" button on Loan Detail and Notification. UPI deep-link requires user to have a UPI app installed. Fallback if no UPI app: copy VPA to clipboard + open bank website. Is UPI the right default or should we show a generic "Marked as Paid" option first?
  • Dark mode — Splash and Onboarding already dark. Full dark mode: dynamic colorScheme via Material You on Android 12+ (adaptive surface colours), manual toggle in Settings for older OS. Implementation cost: Dev to confirm Room + ViewModel layer is display-mode agnostic (should be).
  • Amortization export format — CSV vs. PDF vs. both. CSV is simpler (no iText dependency). PDF looks more professional for user sharing with financial advisor. Both: adds ~1.2MB to APK size.
